For Local Government Clients

Local Government – Attack Surface & OSINT Security Report

自治体様向け Attack Surface & OSINT 技術評価レポート

作成者：Avisail / ADCS
対象：自治体様向け（情報政策・情報システム部門）
対象ドメイン：非開示（自治体公式ドメイン）
本資料は、公開情報と技術的知見に基づき Avisail が独自に実施した評価レポートのひな型です。
実案件では、個別契約に基づき正式な診断レポートとして提供いたします。

1. Executive Summary（概要）

本レポートは、ある自治体の公式ドメイン を対象に、公開情報 （OSINT：Open Source Intelligence）と攻撃者視点の観点から攻撃面（Attack Surface）を整理した サンプル構成です。Active Defense の考え方に基づき、 「どこから何が見えているのか」「メール・Web を中心にどのようなリスクが想定されるか」を 技術的かつ説明責任が果たせる形でまとめています。

1.1 ハイレベル評価

評価項目評価概要
Attack Surface（攻撃面の露出） 中程度 サブドメインは限定的である一方、メール関連設定や過去の運用痕跡により、 攻撃者が足掛かりを得る余地が残されていると評価できます。
Email Security（SPF / DMARC 等） 要改善 SPF レコードは構成されていますが、DMARC ポリシーが p=none のため、 なりすましメール防止の観点では十分な保護が行われていない状態です。
Web / Application 概ね良好 Web サーバヘッダには X-Frame-Options, X-XSS-Protection, X-Content-Type-Options が設定されており、 基本的なセキュリティヘッダは整備されていると判断できます。
Threat Reputation / OSINT 良好 複数のレピュテーション情報源において、対象ドメインは現時点で 悪性としてフラグされておらず、良好な評価になっています。

1.2 主なリスクと提言（要約）

  • 短期 DMARC ポリシーを p=none から quarantine もしくは reject へ段階的に引き上げることを推奨します。
  • 短期 メールドメインに関するサブドメインを整理し、利用実態のないホスト名の廃止・ リダイレクト方針を決定することを推奨します。
  • 中期 OSINT を活用した Attack Surface 管理を年 1〜2 回程度定期的に実施し、 公開情報の棚卸しとダーティデータ（古い証明書・DNS）の削減を行うことを推奨します。
  • 中期 疑似フィッシング訓練を導入し、職員のメール開封・クリック率を可視化した上で、 セキュリティ教育とルール整備を行うことを推奨します。
  • 長期 将来的な 24/7 SOC / MDR 導入を見据え、 ログ基盤およびインシデント初動ガイドの整備を進めることを推奨します。

※ 平たく言えば、「まずはメールの“なりすまし”対策」「次に公開情報の棚卸し」 「最後に本格的な監視体制への橋渡し」という三段階で整理した提言です。

2. Scope & Methodology（対象と方法）

2.1 評価範囲（Scope）

  • 対象ドメイン：非開示（自治体公式ドメイン）
  • 関連サブドメイン：OSINT により複数ドメインを確認済みです。
  • 評価対象：Attack Surface（DNS / Web / Mail）、OSINT によるレピュテーション、 メールセキュリティ設定（SPF / DMARC）です。
  • 除外事項：内部ネットワークスキャン、ログレビュー、脆弱性スキャナによる本格侵入テスト等は 本レポート範囲外としています（公開情報に限定した技術評価構成です）。

2.2 使用プラットフォーム / 手法

Attack Surface / OSINT 系 OSINT

  • インターネット上の公開情報を横断検索する OSINT プラットフォームを利用しています。
  • DNS / 証明書ログ / Passive DNS データベースを参照しています。
  • 自社開発スクリプトによるサブドメイン列挙・集約を実施しています。

Mail / DNS / Web 評価 Mail & Web

  • 標準的な DNS クライアントツールによるレコード確認を行っています。
  • メールセキュリティ診断プラットフォームにより SPF / DMARC / Blacklist チェックを行っています。
  • HTTP レスポンスヘッダ解析ツールにより Web サーバ設定を確認しています。
公開情報の収集・整理はすべて法令・ガイドラインに従い、 対象組織に負荷を与えないパッシブな手法のみを用いて実施しています。

3. Attack Surface Findings（攻撃面の整理）

3.1 サブドメイン構成（例）

ホスト名役割（想定）所見
[ホストA] 公式 Web サイト 代表的なポータルサイト用ホストです。基本的なセキュリティヘッダは付与されており、 一般公開用としては標準的な構成になっていると評価できます。
[ホストB] メール関連ホスト名（現在は実体不明） DNS 上は「名前だけが残っている」状態のホストが存在するケースが見られます。 攻撃者から見ると、過去に利用されていた可能性のあるメールホスト名として フィッシング等に悪用されるリスクがあります。
[ホストC] 外部メール基盤 SPF 設定で参照されているメール送信基盤です。 委託先・運用責任の明確化が重要です。

3.2 公開サービス状況の確認

  • 複数のインターネットスキャン系データベースを確認した結果、 危険性の高い公開サービス（例：インターネットに直接晒された RDP や古い VPN 装置等）は 確認されていません。
  • 一方で、証明書ログや DNS 履歴からは、 過去に利用されていた可能性のあるホスト名が複数確認される場合があり、 今後の運用変更時にも残存しやすい点に留意が必要です。

4. DNS / Email Security（SPF / DMARC 評価）

4.1 SPF レコード

SPF: "v=spf1 ip4:***.***.***.0/26 ip4:***.***.***.236 ip4:***.***.***.217 ip4:***.***.***.19 a:[メール基盤ホスト名] include:[外部メールサービス] -all"

所見

  • SPF 自体は構成されており、基本的な誤設定は見られません。
  • ただし複数の IP アドレスと外部サービスに依存しており、 過去設定が残存している可能性があります。
  • 長期的には、現在実際に利用している送信元 IP / サービスに絞り込み、 不要なエントリを削除することで設定を簡素化することが望ましいです。

4.2 DMARC レコード

DMARC: "v=DMARC1; p=none; rua=mailto:dmarc-report@example.jp"

所見

  • DMARC レコード自体は公開されていますが、ポリシー p=none のため、 DMARC を監視用途にのみ利用している状態です。
  • この状態では、外部の受信側に対し「検査結果に基づいてメールを隔離／拒否してほしい」 という指示が出されないため、なりすましメールの防止効果は限定的です。
  • 段階的に quarantine → reject へ移行するためのログ分析と 試験導入計画を策定することを推奨します。

4.3 MX / Blacklist チェック

  • 主要なスパムブラックリストにおいて、対象ドメインおよび関連 IP は 現時点で悪性として登録されていないことが確認できました。
  • メール関連は県・共同センター等との連携が前提となるケースが多いため、 障害・ブラックリスト登録が発生した際の責任分解点 （自治体・共同センター・委託先）の明確化が重要です。

5. Threat Reputation（脅威レピュテーション）

複数の脅威インテリジェンス／レピュテーション情報源にて、 対象ドメインの評価および Passive DNS 情報を確認した結果、以下のような所見になりました。

5.1 Detection Summary

  • 複数のセキュリティベンダ評価において、対象ドメインは 悪性としてフラグされていません。
  • コミュニティスコア：良好（特段の悪性レポートなし）です。
  • カテゴリ：Government / Legal（自動分類）です。
現時点では、マルウェア配布やフィッシングサイトとしての悪用履歴は 確認されておらず、レピュテーションは良好と評価できます。

5.2 Passive DNS / 履歴

  • 過去数年間で複数の IP に割り当てが行われていますが、 いずれも政府系ネットワークおよび国内ホスティング事業者の範囲内です。
  • 同一 IP 上に悪性ドメインが多数同居しているといった兆候は確認されず、 共有環境としては比較的クリーンな状態と推測されます。

Active Defense の一環として、疑似フィッシングメールを用いた職員向け訓練を実施した場合の レポートイメージを以下に示します（数値はサンプルです）。

項目結果（サンプル）コメント
対象職員数 120 名 庁内の情報系アカウント保有者を対象としたケースです。
メール開封率 42% 件名・送信元が行政機関を装った場合、一定の開封率が見込まれます。

6.1 改善提案（訓練結果にもとづく）

  • 年 1～2 回の疑似フィッシング訓練を継続実施し、傾向を時系列で把握することを推奨します。
  • 管理職向けには「模範的な対応（報告フロー）」を重点的に周知することを推奨します。
  • メールクライアントにおける「差出人表示」「完全な送信元ドメイン表示」を強制する設定を検討することを推奨します。

7. Risk Assessment（リスク評価）

リスクレベル概要
なりすましメールによる住民・職員の誤認 高 DMARC ポリシーが none のため、外部からのなりすましメールが 正規メールとして受信されるリスクが残っています。
過去設定の残存による攻撃面の肥大化 中 利用されていないサブドメインや SPF エントリが残存している場合、 攻撃者にとっての有用な情報となり得ます。
OSINT による情報の先行把握の欠如 中 攻撃者側は OSINT を利用して標的を選定する一方で、 守る側が同等レベルの情報把握をしていない場合、対応が後手に回る可能性があります。
インシデント初動の混乱 中 役割分担・外部ベンダーとの連絡窓口が明文化されていない場合、 初動 24 時間での判断・報告が遅延する可能性があります。

8. Recommendations（推奨事項）

8.1 短期（0〜6ヶ月）

  • DMARC ポリシーを段階的に quarantine → reject へ移行するための 評価・テストを開始することを推奨します。
  • 実際に利用されていないサブドメイン（旧メールホスト名など）の棚卸しを行い、 廃止・リダイレクト方針を決定することを推奨します。
  • 疑似フィッシング訓練の PoC を実施し、セキュリティ教育の企画を進めることを推奨します。
  • 初心者向けに言うと、「まずはメールの“なりすまし”を減らすため、 送信元設定と職員向け注意喚起を最優先で整える段階」です。

8.2 中期（6〜18ヶ月）

  • Attack Surface / OSINT チェックを年次のセキュリティ計画に組み込み、 毎年の「公開情報棚卸し」として定着させることを推奨します。
  • インシデント初動ガイドラインを整備し、庁内の意思決定フロー・報告先・ 外部ベンダー連携方法を明文化することを推奨します。
  • ログ保全方針（どのログを、どこに、どれくらい保持するか）を策定することを推奨します。
  • 初心者向けに言うと、「何か起きたときに“誰が何をするか”を紙に書いておき、 ログも後から追えるように整えておくフェーズ」です。

8.3 長期（18ヶ月〜）

  • 24/7 SOC / MDR ベンダーとの連携を前提としたロードマップを作成することを推奨します。 （県・国・外部委託先との役割分担を整理した上での検討です。）
  • クラウド利用拡大（SaaS / IaaS）を踏まえた Zero Trust / Identity-Centric Security への移行を検討することを推奨します。
  • 初心者向けに言うと、「将来、本格的な監視チームと組むときに困らないよう、 いまから“どの方向に進むかの地図”を作っておく段階」です。

9. Incident First 24 Hours Guide（初動 24 時間ガイド：サマリ）

重大なセキュリティインシデント（情報漏えい・外部攻撃等）が疑われた際の 「最初の 24 時間」で最低限実施すべき事項のサマリです。 詳細版プレイブックは別紙として作成可能です。

  1. 検知と一次報告（0〜1時間）
    庁内窓口への即時連絡、インシデント種別の仮判定、 関係部局・首長への一次報告を行います。
  2. 影響範囲の仮把握（1〜4時間）
    対象システム・アカウント・データ種別の整理、ログの退避、 外部委託先・共同センター等への速報連絡を行います。
  3. 技術的封じ込め（4〜12時間）
    アカウントロック、ネットワーク遮断、該当サービスの一時停止等、 被害拡大防止措置を実施します。
  4. 対外説明方針の検討（12〜24時間）
    住民・報道機関向けの説明草案作成、FAQ 作成、問い合わせ窓口の整理を行います。
